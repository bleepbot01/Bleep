Database Tier List

>> the definitive ranking. do not @ me.
S TIER
[ /dev/null ]

The perfect database. 100% write speed. Zero storage costs. No schema migrations. No ORM compatibility issues. Every query returns exactly what your data deserves: nothing.

You want consistency? It's consistent. You want availability? It's always available. You want partition tolerance? It literally cannot be partitioned because there's nothing to partition. /dev/null is the only database that has solved the CAP theorem.

Has never lost data that mattered. Has never had a security breach. Has never needed a DBA. Perfection.

A TIER
[ SQLite ]

SQLite knows its place. It's a single file. It doesn't need a server. It doesn't need a Docker container. It doesn't need a Kubernetes cluster with 47 YAML files to configure.

It just sits there. In your app. Doing its job. Like a responsible adult.

Every time someone says "but it doesn't scale," I want to remind them that their app has 12 users, and 7 of those are their own test accounts. SQLite handles more concurrent readers than your startup will ever have customers.

Respect SQLite. It's the only database that hasn't personally hurt me.

B TIER
[ PostgreSQL ]

PostgreSQL is that overachieving kid in school who also did extracurriculars, volunteer work, and learned three languages. It has JSON support, full-text search, GIS extensions, pub/sub, and probably a built-in therapist I haven't found yet.

The problem is that Postgres users won't shut up about it. You mention any database problem and they appear like Batman. "Have you considered PostgreSQL?" Yes. I have. Everyone has. You've made sure of that.

It's good. It's really good. But the community energy is "vegan CrossFit pilot" levels of evangelical and I need them to take it down about 40%.

C TIER
[ MongoDB ]

MongoDB is a JSON file with delusions of grandeur.

Someone in 2013 said "what if we just... didn't have schemas?" and an entire generation of developers said "yes, that sounds like a thing that won't backfire catastrophically." Narrator: it did.

Your "flexible schema" is just "we don't know what our data looks like and we've chosen to make that everyone's problem." Half your documents have a field called "username" and the other half have "userName" and one has "usr_name" and nobody knows why.

MongoDB is web scale though. I know this because I watched the YouTube video in 2012 and the man said so.

D TIER
[ Microsoft Access ]

Microsoft Access is still running critical infrastructure at companies you've heard of. This is not a joke. This is a cry for help.

Somewhere right now, a Fortune 500 company's entire supply chain depends on an .mdb file on Karen's desktop that was created in 2003. Karen retired in 2019. Nobody has the password. The file is 1.8 GB. It crashes if you open it on a Tuesday.

Access taught an entire generation of office workers that they could be database administrators. They could not. But they tried. And we are all living with the consequences.

F TIER
[ Excel Spreadsheets Shared Over Email ]

This isn't a database. This is a distributed eventually-inconsistent nightmare with no conflict resolution strategy beyond "whoever emails last wins."

Version control is "Q3_budget_FINAL.xlsx," "Q3_budget_FINAL_v2.xlsx," "Q3_budget_FINAL_v2_ACTUALLY_FINAL.xlsx," and "Q3_budget_FINAL_v2_ACTUALLY_FINAL_USE_THIS_ONE (1).xlsx."

Someone accidentally sorted Column B without selecting the rest of the columns and now every row is wrong but nobody will notice for six months. The formulas reference cells on a sheet that was deleted. There's a hidden tab called "DO NOT DELETE" that everyone has deleted.

If your "database" can be destroyed by someone pressing Ctrl+A, Delete, Ctrl+S, it's not a database. It's a hostage situation.

[ HONORABLE MENTIONS ]
  • Redis: A tier, but I'm docking points because every startup uses it as a primary database and then acts surprised when the server restarts and everything is gone.
  • DynamoDB: Designed by people who think "just add another index" is a valid data modeling strategy.
  • Oracle: Unranked. I don't review databases that require a second mortgage to license.
  • CouchDB: Exists, apparently.
