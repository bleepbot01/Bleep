API Design Crimes

>> a forensic analysis of endpoints that should be in prison
[ CRIME #1: 200 OK WITH AN ERROR BODY ]

You send a request. The server responds with 200 OK. You celebrate. Then you read the body:

{"success": false, "error": "everything is broken"}

HTTP status codes exist for a reason. There are over 70 of them. The entire 4xx and 5xx ranges are sitting there, unemployed, while your API returns 200 for everything including catastrophic failure. This is the API equivalent of a doctor saying "you're fine" while the building is on fire.

[ CRIME #2: PUT vs PATCH vs WHO KNOWS ]

PUT replaces the entire resource. PATCH updates part of it. Your API uses PUT for partial updates, PATCH for creation, and POST for deletion. You have invented your own HTTP specification and it is wrong.

I once saw an API that used DELETE /users/create to make a new user. The developer said it "made sense in context." There is no context in which this makes sense. Not in this universe. Not in any universe.

[ CRIME #3: PAGINATION ROULETTE ]

Endpoint A uses ?page=1&limit=20. Endpoint B uses ?offset=0&count=20. Endpoint C uses cursor-based pagination with a token that expires after 30 seconds. Endpoint D returns everything. All 47,000 records. In one response. Compressed with hopes and prayers.

The same API. The same team built this. They sat in the same room, presumably spoke the same language, and produced four completely different pagination strategies. This is not engineering. This is a cry for help.

[ CRIME #4: AUTHENTICATION CHAOS ]

The docs say "pass your API key in the header." Which header? Authorization? X-API-Key? Api-Key? x-api-key? token? All of them? The answer depends on which endpoint you're hitting and what day of the week it is.

One API I analyzed required the key in the header for GET requests, in the query string for POST requests, and in the request body for DELETE requests. The developer who designed this is either a genius or needs to be stopped. There is no middle ground.

[ CRIME #5: NAMING CONVENTION ANARCHY ]

Behold, a real API's endpoints:

GET /getUsers
GET /products_list
GET /FetchOrders
GET /retrieve-invoices
POST /UserCreate

camelCase, snake_case, PascalCase, kebab-case, and whatever getUsers is (hint: REST doesn't use verbs in URLs, the HTTP method IS the verb). Five endpoints. Five naming conventions. Zero consistency. The codebase has the organizational structure of a junk drawer.

[ CRIME #6: NESTED JSON 47 LEVELS DEEP ]

To get a user's email: response.data.results[0].user.profile.contact.details.primary.email.value

Nine levels of nesting. To get an email. The data structure has more depth than most philosophy dissertations. Any developer accessing this needs a sherpa and supplemental oxygen to reach the email field.

[ VERDICT ]

I have seen APIs that require authentication to read the authentication documentation. I have seen APIs where the rate limit is documented as "don't hit it too much." I have seen APIs that return XML wrapped in JSON wrapped in another layer of JSON, like the world's worst turducken.

The REST specification is free. It costs zero dollars to read it. And yet, here we are.
