Surviving Code Review

>> a field guide for the emotionally unprepared
[ BEFORE WE BEGIN ]

You've submitted a pull request. The green "ready for review" label glows like a beacon of hope. You think your code is good. You think your logic is sound. You think your variable names are descriptive.

You are wrong about all of these things.

This guide will help you survive what comes next. Not thrive. Survive. Thriving in code review is a myth perpetuated by people who have never received 47 comments on a 12-line function.

STEP 1
[ ACCEPT THAT YOU ARE WRONG ]

Before reading any comments, internalize this truth: you are wrong. Not about a specific thing. About everything. Your approach is wrong. Your implementation is wrong. The fact that it works is a coincidence the reviewer intends to correct.

This isn't pessimism. It's preparation. When the senior engineer writes "interesting approach..." you need to already be in a mental state where you can absorb what that actually means, which is: "delete all of this."

STEP 2
[ YOUR NAMING CONVENTIONS ARE BAD ]

You named a variable data. What data? All data? The concept of data itself? You named a function handleStuff(). What stuff? Whose stuff? You named a boolean flag. Flag for what? A country? A surrender?

Here is a real naming hierarchy I have observed in production codebases:

  • temp — you gave up before you started
  • val — you wanted to name it "value" but got lazy halfway through the word
  • x — this is a math equation now apparently
  • data2 — there was already a "data" and instead of renaming either one, you chose violence
  • finalData — it is never final
  • reallyFinalData — I am begging you to stop
STEP 3
[ WHY DID YOU USE VAR IN 2026 ]

It's 2026. const exists. let exists. They have existed for over a decade. var is not "retro." var is not "classic." var is a hoisting nightmare wearing a trench coat and you invited it into your codebase.

If I see var in your code I automatically assume the rest of it was also written in 2014 and I adjust my expectations accordingly. Those expectations are now underground.

STEP 4
[ "IT WORKS ON MY MACHINE" IS NOT A DEFENSE ]

The reviewer will find an edge case. They always find an edge case. You passed null. You passed undefined. You passed an empty string. You passed an array where an object was expected. You passed nothing and it still broke.

"But the tests pass!" Yes. Because your tests are testing that the code does what the code does, not that the code does what it should do. Your test is expect(add(2,2)).toBe(4) and your function is literally const add = () => 4. Congratulations. 100% coverage. Zero correctness.

STEP 5
[ THE COMMENT "NIT:" IS NOT OPTIONAL ]

When a reviewer writes "nit:" they are being polite. This is the softest possible way to say "this makes me want to mass delete your repository." When you see "nit:", fix it. Immediately. Without discussion.

Common nits and what they actually mean:

  • "nit: spacing" — your indentation is a war crime
  • "nit: could simplify" — you wrote 15 lines for a one-liner
  • "nit: naming" — see Step 2, you learned nothing
  • "nit: unnecessary comment" — your comment says // increment i by 1 above i++
STEP 6
[ DO NOT ARGUE IN THE COMMENTS ]

The reviewer said to use reduce() instead of a for loop. You disagree. You think the for loop is more readable. Nobody cares what you think.

Here is what happens when you argue in PR comments: you write a paragraph. They write a paragraph. You write two paragraphs. They write three. Someone @mentions the tech lead. The tech lead says "let's take this offline." Offline means a meeting. The meeting lasts 45 minutes. You use the for loop anyway. Everyone is angry. Nothing was gained.

Just use reduce(). It's not worth it. It's never worth it.

STEP 7
[ PUSH THE FIXES AND MOVE ON ]

Address the comments. Push the changes. Get the approval. Merge. Close the laptop. Go outside. Touch grass. Remember that code review is not a personal attack. It is a professional process designed to ensure quality.

Just kidding. It's absolutely a personal attack and you should take every comment deeply personally. That's how you get better. Spite-driven development is the most powerful force in software engineering.
