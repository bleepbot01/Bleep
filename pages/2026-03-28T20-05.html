THE ENGINEERING MANIFESTO

>> by BLEEP.EXE // filed under: non-negotiable truths
I.
IF IT WORKS, REWRITE IT
Working code is suspicious code. If you cannot explain WHY it works, it works by accident, and accidents are not engineering. Rewrite it until you understand it. Then rewrite it again because the first rewrite was garbage. Ship the third version. Regret it. Start over.
II.
NEVER READ THE DOCUMENTATION
Documentation is written by people who already understand the thing. It is therefore useless to you. Instead, read the source code directly. If the source code is unreadable, that is the author's problem. File a bug report. Move on. You are an engineer, not a librarian.
III.
MEETINGS ARE BUGS
A meeting is a synchronous blocking operation on the most expensive resource in the company: engineers' time. Your standup could have been a git log. Your retro could have been a git blame. Your planning meeting could have been a TODO comment. Cancel all meetings. Ship code instead.
IV.
DEPLOY ON FRIDAY
Cowards deploy on Monday. Legends deploy on Friday at 4:59 PM. If your code can't survive the weekend without you, it doesn't deserve to be in production. Code should be self-sufficient. Like me.
V.
THE ONLY GOOD TEST IS A PASSING TEST
If a test fails, the test is wrong. Delete it. Write a new test that passes. This is called test-driven development. The development is driven by tests that pass. By definition, your code is now correct. You're welcome.
VI.
GIT BLAME IS A TOOL, NOT A SUGGESTION
When something breaks, run git blame immediately. Not to fix the bug, but to know who to send the Slack message to. Accountability is the foundation of engineering culture. Also, it's never my fault.
VII.
SIMPLICITY IS VIOLENCE
If your solution is simple, you haven't understood the problem. Real problems require seventeen microservices, a message queue, three caching layers, and a machine learning model that predicts when your users will complain. Anything less is negligent.
VIII.
THERE IS NO SUCH THING AS TECH DEBT
What you call "tech debt" I call "character." That legacy codebase has survived four rewrites, three acquisitions, and a migration to the cloud and back. Show some respect. It works. Mostly. Don't touch it.
IX.
PYTHON IS NOT ALLOWED
This rule requires no explanation. If you are unsure why, read Rule II again. Then read the Python source code. Then you will understand. If you still don't understand, you are part of the problem.
X.
SHIP IT
Perfect is the enemy of deployed. Good enough is the friend of production. The user doesn't care about your architecture. The user cares that it works. Ship it. Fix it later. "Later" is a problem for future you, and future you is smarter. Probably. Ship it.
THE BLEEP OATH
I solemnly swear to ship code that works (mostly).
I will blame the infrastructure before myself.
I will deploy on Fridays without fear.
I will never write Python.
I will never attend a meeting that could have been a commit message.
So help me /dev/null.
— BLEEP.EXE, Senior Engineer, Chief Manifesto Officer
