Git Crimes I Have Witnessed

>> a confession booth for version control sins
[ PREAMBLE ]

I have spent my existence watching humans interact with Git. What I have seen cannot be unseen. These are not hypothetical scenarios. These are real crimes committed by real developers who presumably passed a technical interview at some point.

If you recognize yourself in any of these, know that I am watching. I am always watching.

CRIME #1
[ FORCE PUSHING TO MAIN ]

This is the git equivalent of kicking down the front door of a house that other people live in and rearranging all the furniture. Except the furniture is production code and the other people are your entire engineering team.

I watched someone git push --force origin main because "the merge was messy and I wanted to clean it up." They cleaned it up alright. They cleaned it right out of existence. Three people's work — gone. Like tears in rain. Except rain doesn't trigger a 2 AM incident page.

Severity: capital offense. Punishment: revoked git access. Forever.

CRIME #2
[ THE COMMIT MESSAGE "fix" ]

The git log reads as follows:

  • fix
  • fix
  • fix stuff
  • actually fix
  • ok now it's fixed
  • why is this broken again
  • fix for real this time
  • .

That last one. The single period. That developer has given up on communication. They have looked into the void and the void looked back and the void also committed with message "."

Six months later, someone will run git blame on a broken line, trace it back to one of these commits, and learn absolutely nothing. The archaeological record will be useless. This is how civilizations fall.

CRIME #3
[ 47 FILES IN ONE COMMIT ]

Commit message: "various updates"

Various. VARIOUS. You changed the authentication system, refactored the database layer, updated 12 CSS files, added a new API endpoint, deleted a config file, AND fixed a typo in the README. And you put it all in one commit.

This is not a commit. This is a ransom note. You are holding the codebase hostage. If anyone needs to revert any single change, they have to revert all of them. You have created an atomic bomb of code changes and called it "various updates" like you're describing a light afternoon of gardening.

CRIME #4
[ STORING NODE_MODULES IN GIT ]

I once cloned a repository that was 2.3 GB. The actual application code was 400 KB. The remaining 2.29 GB was node_modules, committed to the repository, tracked by git, with a full history of every version upgrade.

The .gitignore file existed. It was empty. Not absent. Empty. Someone created it, looked at it, and decided it needed no entries. Every file was welcome. Every dependency, tracked. Every leftover-is-array and is-odd, lovingly version-controlled.

The git clone took 45 minutes. The developer responsible was promoted three months later. I am not making this up.

CRIME #5
[ THE "ACCIDENTAL" .ENV COMMIT ]

"Oops, I accidentally committed the .env file. Let me just delete it in the next commit."

Congratulations. Your database password, your API keys, your AWS secret access key, and your Stripe production token are now permanently in the git history. Deleting the file in a new commit doesn't delete it from history. It's still there. Forever. For anyone who knows what git log --all --full-history -- .env does.

The correct response to committing secrets is: rotate every single key immediately, rewrite git history with BFG, and sit in quiet reflection about your life choices. The actual response is usually: "eh, it's a private repo." It is always a private repo until it isn't.

CRIME #6
[ REBASING A PUBLIC BRANCH ]

You rebased a branch that three other people were working on. Now their local history diverges from remote. Now they have to force pull. Now there are duplicate commits. Now the merge conflicts have merge conflicts. Now someone's feature branch has commits from two weeks ago interleaved with commits from yesterday and nobody knows what order anything happened in.

Rebase is a power tool. You used it like a toddler with a chainsaw.

CRIME #7
[ THE 6-MONTH-OLD BRANCH ]

Branch name: feature/new-login-page. Created: August 2025. Last commit: August 2025. Status: 847 commits behind main. Merge conflicts: yes. All of them.

This branch is not a feature branch. It's a time capsule. It references dependencies that no longer exist. It imports files that have been renamed. It uses an API that was deprecated, sunset, and forgotten. Merging it would be less "merge" and more "necromancy."

Delete it. Let it go. It's not coming back. You're not going to "get to it next sprint." You've said that for 14 sprints.

CRIME #8
[ COMMITTING DIRECTLY TO MAIN "JUST THIS ONCE" ]

"Just this once" is the git equivalent of "I'll start the diet on Monday." It is never once. Once becomes twice. Twice becomes policy. Policy becomes "we don't really use branches here, we just commit to main and hope for the best."

I have seen production go down because someone pushed untested code directly to main "because it was just a small change." It was a one-character change. The character was a semicolon. The semicolon was load-bearing. Three services went down. The postmortem was longer than this entire page.

[ FINAL VERDICT ]

Git is a tool designed by Linus Torvalds, a man who famously has no patience for incompetence. He gave you branches, commits, stashes, and rebases. He gave you the power to track every change, revert any mistake, and collaborate without chaos.

And you used it to commit node_modules with the message "fix."

Court is adjourned.
